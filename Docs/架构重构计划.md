# Panasonic Smart China HA 插件架构重构计划

## 目标边界

本轮重构从 `main` 分支重新开始，不采用 `feature/account-based-devices` 分支的实现。该分支只作为历史实验参考，用来识别账号级模型需要解决的问题，不作为 merge 或 cherry-pick 的来源。

项目当前的真实适配范围限定为中国区「松下智能家电」App 下的 0900 品类风管机设备。后续扩展更多品类和设备型号时，必须建立在 0900 控制链路已经稳定、可验证、可回归的基础上。

核心目标有三类：

1. 健壮化现有 0900 风管机控制链路，减少误更新、误覆盖和静默失败。
2. 将 HA 配置模型从「一个设备一个配置项」升级为「一个账号一个配置项，账号下挂载多个设备」。
3. 抽象设备 profile / adapter 层，为未来融合其他品类适配代码留下明确扩展点。

## 当前基线判断

当前 `main` 代码可以作为重构起点，因为它的真实运行面较小，主要集中在：

- `custom_components/panasonic_smart_china/config_flow.py`
- `custom_components/panasonic_smart_china/climate.py`
- `custom_components/panasonic_smart_china/const.py`
- `custom_components/panasonic_smart_china/__init__.py`

但它存在几个需要优先处理的结构性问题：

- API 调用、登录、token 生成、设备 profile、HA entity 状态解析混在一起。
- SSID 作为单设备 entry data 固化，和松下账号单点登录机制冲突。
- 命令发送后不校验服务端响应，容易把失败控制显示成成功。
- Read-Modify-Write 在读取失败时仍可能使用旧缓存继续写入。
- 配置流以设备为粒度创建 entry，未来无法自然表达账号、家庭、设备列表和重新扫描。
- 已出现未充分验证的其他品类逻辑，容易影响 0900 主链路稳定性。

## 不采用 feature/account-based-devices 的原因

`feature/account-based-devices` 的方向是正确的：它尝试把 config entry 改成账号级，并在一个 entry 下创建多个设备实体。但它的问题是没有真正拆开底层职责，仍然把旧的设备控制逻辑包在账号模型里：

- API client、session、profile、entity 仍然耦合。
- 账号 options flow 不完整，无法可靠支持重新扫描、启用/禁用设备、更新设备配置。
- migration 逻辑较激进，试图合并并移除旧 entry，风险较高。
- 未验证品类仍然进入主路径。
- SSID 更新与已初始化实体之间的 reload 语义不够清晰。

因此后续实现应基于 `main` 新建分支，从较小、可验证的提交开始重构。

## 目标架构

建议最终形成以下模块边界：

```text
custom_components/panasonic_smart_china/
  __init__.py              # 集成入口，设置账号级 runtime data，转发平台
  const.py                 # domain、常量、配置 key
  api.py                   # 松下云 API client，负责 HTTP 请求和错误解析
  auth.py                  # 登录、session、reauth 相关数据模型
  token.py                 # device token 生成算法
  models.py                # AccountInfo、PanasonicDevice、DeviceConfig 等结构
  profiles/
    __init__.py            # profile 注册表和 lookup
    ducted_ac_0900.py      # 0900 风管机 profile
  climate.py               # HA ClimateEntity 适配层，只做 HA entity 映射
  config_flow.py           # 账号登录、设备选择、账号级 entry 创建
  options_flow.py          # 重新扫描、设备启用、设备配置更新
```

初期可以不一次性拆出所有文件，但实现方向必须保证：

- API 层不知道 HA entity。
- profile 层不知道 config flow。
- climate entity 不直接拼登录请求，不直接实现 token 算法。
- 账号 runtime data 是实体共享的 source of truth。

## 账号级 HA 模型

未来 HA 中应该表现为：

- 一个 Panasonic 账号对应一个 `ConfigEntry`。
- 账号 entry 下挂载多个 Panasonic 设备。
- 每个真实设备在 HA device registry 中是独立 device。
- 每个设备按能力创建一个或多个 entity。当前 0900 只创建 climate entity。

建议 entry data 结构：

```python
{
    "username": "...",
    "usrId": "...",
    "SSID": "...",
    "familyId": "...",
    "realFamilyId": "...",
    "devices": {
        "<deviceId>": {
            "deviceName": "...",
            "category": "0900",
            "profile": "ducted_ac_0900",
            "token": "...",
            "enabled": True,
            "sensor_entity_id": "sensor.xxx"
        }
    }
}
```

建议 runtime data 结构：

```python
hass.data[DOMAIN][entry.entry_id] = {
    "client": PanasonicApiClient(...),
    "account": AccountInfo(...),
    "devices": {...}
}
```

账号 entry 的 `unique_id` 应使用稳定账号标识，优先使用登录后返回的真实 `usrId`。设备 entity 的 `unique_id` 应使用 `deviceId` 加 entity 类型，例如：

```text
panasonic_smart_china_<deviceId>_climate
```

## 第一阶段：稳住 0900 风管机基础链路

目标：不扩品类，不做复杂迁移，只让当前真实设备控制更可靠。

任务：

- 抽出 `token.py`，为现有 0900 device token 算法增加单元测试或最小验证脚本。
- 抽出 `api.py`，统一封装：
  - `login()`
  - `get_devices()`
  - `get_ac_status()`
  - `set_ac_status()`
- API 层统一处理：
  - HTTP status 非 200。
  - JSON 解析失败。
  - 松下业务错误码。
  - SSID 过期错误。
  - 超时。
- `climate.py` 命令发送后必须检查服务端结果，失败时不做乐观状态更新。
- `async_added_to_hass()` 后立即拉取一次真实状态，避免默认显示关机/26 度。
- 增加 `available` 状态，连续或关键状态拉取失败时让实体不可用。
- Read-Modify-Write 失败策略收敛：
  - 状态读取成功时，用最新状态合并指令。
  - 状态读取失败且无缓存时，中止写入。
  - 状态读取失败但有缓存时，仅允许明确安全的局部命令，危险命令应中止。
- 只保留 0900 风管机主路径；未验证品类不要暴露在正式配置流中。

验收标准：

- 现有 0900 设备可登录、发现、添加、读取状态、开关机、切模式、调温、调风速、切静音。
- 云端返回失败时 HA 不显示为控制成功。
- SSID 过期能进入 reauth 或至少明确报错。
- 没有新增未验证品类行为。

## 第二阶段：账号级配置模型

目标：从设备级 entry 切换到账号级 entry。

任务：

- 新建基于 `main` 的开发分支，例如：

```bash
git checkout main
git checkout -b codex/account-based-rebuild
```

- `config_flow.py` 改为：
  - 输入账号密码。
  - 登录并获取 `usrId / SSID / familyId / realFamilyId`。
  - 获取账号下设备列表。
  - 只展示支持的 0900 设备。
  - 允许选择一个或多个设备。
  - 创建一个账号级 entry。
- `__init__.py` 建立账号级 runtime data，并转发 `climate`。
- `climate.py` 从账号 entry 的 `devices` map 创建多个 entity。
- 每个 entity 共享同一个账号 API client。
- reauth 成功后更新账号 entry data 并 reload entry。
- options flow 支持：
  - 重新扫描账号设备。
  - 启用新发现的支持设备。
  - 禁用已配置设备。
  - 修改单设备的外部温度传感器。

验收标准：

- 一个账号只创建一个 integration entry。
- 一个账号 entry 下可挂载多台 0900 设备。
- 重新登录后所有设备共用新 SSID。
- 添加第二台设备不会触发重复登录踢掉第一台设备。

## 第三阶段：profile / adapter 扩展层

目标：为未来融合其他人的适配代码建立稳定接口。

建议 profile 至少描述：

```python
DeviceProfile(
    id="ducted_ac_0900",
    category_ids={"0900"},
    device_type="climate",
    name="松下风管机 0900",
    status_endpoint="ACDevGetStatusInfoAW",
    set_endpoint="ACDevSetStatusInfoAW",
    token_strategy="sha512_device_id",
    temp_scale=2,
    features={...},
    safe_status_keys={...},
)
```

每个 profile 或 adapter 需要提供：

- `matches(device)`：判断是否支持设备。
- `parse_status(raw)`：把松下状态转为内部状态。
- `build_command(action, current_state)`：从 HA action 构造松下 payload。
- `validate_response(response)`：判断控制是否真正成功。
- `supported_features()`：返回 HA entity 能力。

融合外部适配代码时，应先转成 profile/adapter，再进入实体层。不要直接把不同品类的字段和接口写进 `climate.py`。

## 第四阶段：旧配置迁移

迁移应最后做，不能和账号模型重构一起冒进。

建议策略：

- 第一版账号级模型可以先不自动迁移，要求用户重新添加账号。
- 如果要迁移，先支持单个旧设备 entry -> 新账号 entry。
- 多个旧 entry 合并为一个账号 entry 需要单独验证。
- 迁移过程中不要静默删除旧 entry，除非有明确测试覆盖和回滚策略。

## 分支与提交策略

后续开发只基于 `main` 创建新分支，不继续使用 `feature/account-based-devices`。

建议分支：

```bash
codex/account-based-rebuild
```

建议提交顺序：

1. `docs: add architecture rebuild plan`
2. `refactor: extract panasonic api client`
3. `refactor: extract token generation`
4. `fix: harden ducted ac command response handling`
5. `feat: introduce account runtime data`
6. `feat: create climate entities from account devices`
7. `feat: add account options flow`
8. `feat: add reauth support`

每一步都应保持 0900 风管机主链路可运行。

## 暂不处理的事项

- 暂不接入浴霸或其他未验证品类。
- 暂不自动迁移旧设备级配置。
- 暂不引入复杂 coordinator，除非账号级多设备轮询已经需要统一调度。
- 暂不把 README 作为架构文档，README 只在实现稳定后更新用户文档。

- 第四阶段决策变更：2.0 作为不兼容升级发布，不做 1.x 自动迁移。用户需要删除旧集成并重新添加账号，旧 entity 和自动化引用需由用户自行调整。
